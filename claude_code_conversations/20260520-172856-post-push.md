## レビュー結果

### 1. 変更の意図
`【ここから有料部分｜500円】` のような有料マーカーから抽出した価格文字列を `parseInt` でサニタイズする修正。自然な防御的コーディングで、意図は明確。

### 2. リスク

**軽微だが確認が必要な点：**

- `parseInt(rawPrice, 10) || parseInt(DEFAULT_PAID_PRICE, 10)` の `||` フォールバックは、価格が `0` のときも `DEFAULT_PAID_PRICE`（100円）に上書きされる。0円の有料記事は存在しないので実害はないが、意図せぬ挙動になり得る
- `DEFAULT_PAID_PRICE = "100"` が文字列なので `parseInt("100", 10)` は毎回走る。定数を数値 `100` にしておくとより堅牢

### 3. Note/X/Threads への影響
影響なし。`preparePaidArticleBody` は `generate.js` の有料Note投稿パス限定の関数で、X/Threads/通常Note投稿には触れていない。

### 4. 次にCodexへ依頼すべき改善

優先度順：

1. **`DEFAULT_PAID_PRICE` を文字列から数値定数に変更**（`"100"` → `100`）し、`parseInt` 呼び出しを1回に整理。小さいが一貫性が増す
2. **price が Note APIの許容範囲内か検証を追加**（例: 100〜50000円のバリデーション）。不正な値がAPIに渡ったときのエラーが現状サイレントに起きる可能性がある
3. **`gen-premium-article.js` 側の同様の価格処理を確認**し、同じサニタイズが必要かどうか横断チェックする
