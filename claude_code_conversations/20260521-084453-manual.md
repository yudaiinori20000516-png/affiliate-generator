## 変更内容の確認レポート

### 1. 変更の意図（自然か）

**意図は明確で合理的。** 以下3つのリファクタリングがまとめて行われた：

- **`paid-note.js` 新設**：`generate.js` にベタ書きされていた `preparePaidArticleBody` と価格定数を共通モジュールに抽出
- **`toPublicNoteUrl()` 追加**：Note投稿後のURLを `https://note.com/ailab2026/n/nXXX` 形式に正規化。投稿完了後のpage.url()が意図しないURL（編集画面等）になる場合への対策
- **有料設定の失敗をthrowに変更**：`gen-premium-article.js` で「有料ラジオボタンが見つからない→サイレントにスキップ」だったのを、**見つからなければ即エラー停止**に変更。無料で誤公開されるリスクを排除

投稿順序も `X → Note → Threads` から **`Note → X → Threads`** に変更されている。NoteのURLをX/Threadsに使う流れと整合。

---

### 2. 運用上のリスク

| リスク | 評価 |
|---|---|
| `validatePaidPrice` の二重呼び出し | **軽微**。`preparePaidArticleBody` 内でも `validatePaidPrice` を呼ぶが、`isPaid=false` のとき（マーカーなし）は `DEFAULT_PAID_PRICE=100` でバリデーションが走る。無料記事のつもりでも `isPaid=false` で通過するので実害なし |
| `toPublicNoteUrl` のパターン漏れ | **要確認**。正規表現は `/notes/(nXXX)/` と `/n/(nXXX)/` の2パターンのみ。NoteがURLスキームを変えた場合は元URLを返すフォールバックがあるので破綻しないが、X/Threadsに編集URL等が流れる可能性は残る |
| `NOTE_ACCOUNT` のenv依存 | `process.env.NOTE_ACCOUNT || "ailab2026"` でデフォルト埋め込み済みなので問題なし |
| 有料設定throwへの変更 | **意図的な強化**。ただし本番でNote側のUI変更があった場合に全投稿が止まるリスクがある |

---

### 3. Note/X/Threads 投稿フローへの影響

- **投稿順序変更** が実質的な変更点。Note URL取得 → X投稿 → Threadsに渡す、という流れになり**Threads側がnote URLを受け取れる設計と一致**した（以前はXを先に投稿していたため順序の不整合があった）
- `generate.js` 側の `preparePaidArticleBody` は外部モジュールからのimportに変わったが、ロジック自体は同一なので既存の無料記事フローには影響なし

---

### 4. Codexへの次の依頼候補

優先度順：

1. **`toPublicNoteUrl` のテスト追加**（`paid-note.js` のunit test）：現状ゼロテストなので、URLパターンが変わったときの検知手段がない
2. **`preparePaidArticleBody` の `isPaid=false` 時の `validatePaidPrice` 呼び出しを除去**：マーカーなし記事でも価格バリデーションが走るのはノイズ。`isPaid` が `false` なら `price` は使わないので `validatePaidPrice` を呼ぶべきでない
3. **`gen-premium-article.js` の `PRICE` をenv変数化**：現在 `validatePaidPrice(500)` とハードコード。`PAID_PRICE=980 node gen-premium-article.js` で動的に変えられるようにすると運用しやすい
4. **Note有料設定失敗時のスクリーンショット保存**：throwで止まるが、どのUI状態で失敗したか分からない。Playwright の `page.screenshot()` をcatch内に追加するとデバッグが容易
