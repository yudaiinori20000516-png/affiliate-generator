## 変更レビュー

### 1. 変更の意図は自然か

**自然。** 前回コミット（`e4ac412` Harden, `544d451` Sanitize）に続く価格バリデーション強化の流れ。今回は「範囲外の価格でNote投稿が走ってしまう」リスクを `throw` で明示的にブロックするようにした。型も `string→number` に統一されており、意図は一貫している。

---

### 2. 運用上のリスク

**潜在リスクあり（低〜中）：**

- **`parseInt` の失敗ケースが二重に走る**  
  `rawPrice = parseInt(...)` が `NaN` になると、次行の `Number.isInteger(NaN)` は `false` → `DEFAULT_PAID_PRICE(100)` にフォールバック。ただしその後 `isPaid` が `true` なら `100 >= 100` でバリデーション通過するので今は問題ない。意図通り動く。

- **`isPaid && price < MIN` のガード順序**  
  `isPaid` が `false`（マーカーなし記事）のときはバリデーションをスキップする仕様。これは正しい。

- **`MAX_PAID_PRICE = 50000` の根拠が不明**  
  Noteの実際の有料記事上限額（現在 **9,999円**）と乖離している。AIが50,000円の記事を生成してもここをすり抜けてしまう。

---

### 3. Note/X/Threads フローへの影響

- **Note投稿フロー**：`preparePaidArticleBody` で `throw` が起きると `postToNote` まで到達しない。X・Threads はその前に完了している場合、SNS投稿だけ行われてNoteが欠落するリスクがある（フロー設計上の既知課題）。今回新たに悪化はしていない。
- **X・Threads**：直接影響なし。
